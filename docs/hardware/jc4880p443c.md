# Board: JC4880P443C_I_W

GUITION/JCZN 4.3" ESP32-P4 display board. Core module `JC-ESP32P4-M3`
(ESP32-P4 + ESP32-C6 companion SoC for Wi-Fi/Bluetooth).

## Main controller and memory (Imported)

- SoC: ESP32-P4, dual-core RISC-V.
- 768 KB HP L2MEM, 32 KB LP SRAM, 128 KB HP ROM.
- PSRAM: 32 MB. Flash: 16 MB.

## Wireless (Confirmed here)

See [docs/validation/wifi-hosted-link-bring-up.md](../validation/wifi-hosted-link-bring-up.md)
and [docs/decisions/0001-wifi-connectivity.md](../decisions/0001-wifi-connectivity.md).

- ESP32-P4 has no native Wi-Fi/Bluetooth; connectivity goes through the
  ESP32-C6 companion SoC via ESP-Hosted over SDIO.
- Do not assume direct `esp_wifi_*` calls work on the P4 without the C6 link
  configured.
- **The ESP32-C6 has its own separate flash and keeps its own Wi-Fi
  association state independently of the ESP32-P4.** `idf.py erase-flash`
  only erases the P4's flash. If this physical board was previously used
  with another project (e.g. `siswood-air-quality`), the C6 may already be
  connected to a remembered network on boot, before this project's own
  Wi-Fi code ever calls `esp_wifi_connect()`. Don't assume STA always
  starts in a clean disconnected state on this board.

## Display (Confirmed here)

See [docs/validation/display-touch-bring-up.md](../validation/display-touch-bring-up.md):
panel initializes cleanly and the LVGL label is visually confirmed on the
physical screen (black text on white background, centered).

- Panel: 4.3" TFT IPS, 480x800.
- Pixel format: RGB888 / 24 bits per pixel (the sibling project's own
  hardware-facts summary previously said RGB565; cross-checking its actual
  panel/LVGL config code shows `LCD_COLOR_FMT_RGB888` /
  `LV_COLOR_FORMAT_RGB888` at 24 bpp is what actually initializes
  successfully — the code, not the earlier prose summary, is the source of
  truth here).
- Driver IC: ST7701S over MIPI DSI, 2 lanes, 500 Mbps lane bitrate.
- DPI timing: clock 34 MHz; horizontal active 480 / back porch 42 / pulse
  width 12 / front porch 42; vertical active 800 / back porch 8 / pulse
  width 2 / front porch 166. Frame buffers: 1.
- Backlight GPIO: `GPIO_NUM_23`. LCD reset GPIO: `GPIO_NUM_5`.
- **The default init command sequence in the registry `esp_lcd_st7701`
  component does not work reliably on this board.** A board-specific ST7701
  init sequence is required (passed via `st7701_vendor_config_t.init_cmds`).
- Do not use the registry `ST7701_480_360_PANEL_60HZ_DPI_CONFIG()` macro —
  it does not match this board's timing. Build an explicit
  `esp_lcd_dpi_panel_config_t` instead.
- **Conflicting vendor source found:** the vendor's own official ESP-IDF
  demo (`JC4880P443C_I_W/1-Demo/idf_examples/ESP-IDF/lvgl_demo_v9`, using
  `espressif__esp32_p4_function_ev_board`) ships with `CONFIG_BSP_LCD_TYPE_1024_600=y`
  selected for this panel, and *does* use the registry default
  `ST7701_480_360_PANEL_60HZ_DPI_CONFIG()` macro with `lane_bit_rate_mbps = 750`
  and no custom init command table (it's present in source but commented
  out). This project follows the 34 MHz / 42-12-42 / 8-2-166 / 500 Mbps /
  custom-init-sequence configuration instead, because that is the one with
  an explicit "started successfully" runtime observation behind it — the
  vendor demo's own working status was not independently confirmed here.
  If display bring-up in this project fails, the vendor demo's config is
  the documented fallback to try.

## Touch (Confirmed here)

See [docs/validation/display-touch-bring-up.md](../validation/display-touch-bring-up.md):
GT911 detected on I2C and registered as an LVGL input device
(`TouchPad_ID:0x39,0x31,0x31`). Actual touch *events* (finger contact) not
yet exercised — only controller detection and registration are confirmed.

- Controller: GT911, I2C.
- I2C pins: SDA `GPIO_NUM_7`, SCL `GPIO_NUM_8`.
- Reset/interrupt: `GPIO_NUM_NC` in the known-working 480x800 configuration
  (i.e. not wired to a dedicated GPIO in that setup).

## Known pitfalls (Imported)

- PSRAM must run at 200 MHz (`CONFIG_SPIRAM_SPEED_200M=y`). At 20 MHz this
  produced `lcd.dsi.dpi: can't fetch data from external memory fast enough,
  underrun happens`. This project's `sdkconfig.defaults` already sets
  200 MHz.
- This MIPI DPI panel does not support hardware rotation:
  `esp_lcd_panel_swap_xy(...): swap_xy is not supported by this panel`.
  Keep `.swap_xy = false` and `.sw_rotate = true` in the LVGL port config.
- DMA2D has triggered an ISR assert on this board on ESP-IDF 5.5.x with the
  registry ST7701 path. Keep `.flags.use_dma2d = false` until revalidated.
- A `ledc: GPIO 23 is not usable, maybe conflict with others` warning during
  backlight PWM setup is a separate, lower-priority issue from DSI underrun —
  if the panel is otherwise lit, treat brightness/PWM as a follow-up.
- Two USB-C ports exist (full-speed and high-speed); flashing/serial
  behavior can differ between them. If flashing fails, try the other port
  and note in the validation report which port was used for what.

## Recommended component dependencies (Imported)

```yaml
dependencies:
  espressif/esp_lcd_st7701:
    version: "^2.0.2"
    public: true
  espressif/esp_lcd_touch_gt911:
    version: "^1"
    public: true
  espressif/esp_lvgl_port:
    version: "^2"
    public: true
  espressif/esp_codec_dev:
    version: "1.2.*"
    public: true
```

LVGL: prefer v9. With RGB888 (24bpp), `esp_lvgl_port` rejects a DMA draw
buffer (`buff_dma = true` requires RGB565 — verified by this project's own
build, see `docs/validation/`). Use `buff_dma = false`, `buff_spiram = true`
instead.
